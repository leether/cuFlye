# Task Card: cuFlye M3e Sampled Pack Batch Planner

Status: active

Created: 2026-06-30

Owner: cuFlye maintainers

Remote: https://github.com/leether/cuFlye

## Intent

Add a reproducible sampled-pack fixture generator and batch planner so the M3d
worker arena/device-prefix path can be measured across heterogeneous real-read
shapes instead of only repeated identical requests.

The core question this card must answer is:

```text
Can cuFlye plan and execute multiple sampled real-pack requests in one worker
process while preserving candidate-record-v1 equivalence for every request?
```

## Background

M3d proved repeated stable-shape worker reuse:

- M3d warm backend total before JSON: `18.643 ms`
- M3d warm device allocation: `0.001 ms`
- M3d warm arena allocations: `0`
- M3d warm arena reuses: `7`
- M3d candidate diff: `match`

The current DGX scratch state only has one independent real Flye pack-dump-v0
query (`query_neg253`). Capturing arbitrary later Flye queries by letting graph
logic advance would weaken the fail-closed boundary. M3e therefore uses
deterministic sampled real-read windows derived from that real pack and records
this limitation explicitly.

## In Scope

- Add a tool that can derive multiple sampled pack-dump-v0-compatible fixtures
  from one real pack directory.
- Add a tool path that emits a worker `requests.jsonl` plus a compact batch
  plan JSON with estimated bytes and shape metadata.
- Preserve existing M3d worker behavior.
- Validate every sampled worker output against a CPU oracle generated by the
  read-window backend.
- Record compact DGX proof under `tests/golden/`.

## Out of Scope

- No Flye graph or overlap behavior change.
- No new in-process CUDA inside Flye.
- No claim that multiple independent Flye query ids were captured.
- No full assembly speedup claim.
- No large TSV outputs committed.

## C++/CUDA Style Constraints

- Do not touch CUDA kernel semantics in this card.
- Keep existing C++/CUDA resource ownership unchanged.
- Generated sampled packs are proof artifacts only and must not be committed.
- Batch planning must fail closed when requested sample lengths or budgets are
  invalid.

## Deliverables

- `tools/plan_cuda_worker_batch.py` for sampled pack generation and worker
  request planning.
- M3e DGX proof under `tests/golden/`.
- Updated ROADMAP, golden manifest index, and this Task Card after proof.

## Acceptance Gates

- Tool rejects invalid sample lengths and missing pack files.
- Tool emits at least three sampled pack requests from the real M2b pack.
- Worker builds on DGX with `nvcc`.
- Existing read-window smoke build still succeeds.
- CPU oracle output validates as candidate-record-v1 for every sampled pack.
- Worker output validates as candidate-record-v1 for every sampled pack.
- Worker output canonical diff against the sampled CPU oracle is `match` for
  every request.
- At least one warm sampled request reports `worker_device_arena_reuses > 0`.
- At least one sampled request has warm backend total below its CPU oracle time.
- Batch plan reports zero CPU fallback requests.
- No large TSV output is committed.
